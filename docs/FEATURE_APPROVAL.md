# 功能审批流程 | Feature Approval Process

> 版本: 0.1 | 创建: 2026-06-18

## 流程概述

任何新增功能必须通过以下5个问题的审批：

## 审批问题

### Q1: 它是否强化 Legacy Loop？

**定义**: Legacy Loop = 行动 → 遗产 → 死亡 → 后果 → 再行动

**判断标准**:
- 玩家的行为是否能产生可被世界继承的遗产？
- 遗产是否会在后续世代产生可感知的后果？
- 后果是否能驱动玩家开启下一世？

**如果否**: 直接拒绝开发。

**如果不确定**: 需要设计文档说明如何强化Legacy Loop。

---

### Q2: 玩家是否能感知？

**定义**: 玩家能否在游戏中明显感知到这个功能的存在和效果？

**判断标准**:
- 是否有明确的视觉/听觉/文字反馈？
- 是否影响玩家的决策？
- 是否影响NPC的行为？
- 是否影响世界的状态？

**如果否**: 需要重新设计或降低优先级。

---

### Q3: 是否能在Steam截图中体现？

**定义**: 这个功能是否能成为Steam商店页面的卖点？

**判断标准**:
- 是否能在截图中展示？
- 是否能在一句话中描述？
- 是否能吸引玩家点击？

**如果否**: 不影响开发，但不能作为主要卖点。

---

### Q4: 是否增加维护成本？

**定义**: 这个功能是否增加了未来维护的复杂度？

**判断标准**:
- 是否需要额外的代码维护？
- 是否需要额外的内容制作？
- 是否需要额外的测试？
- 是否影响现有系统的稳定性？

**如果否**: 可以考虑开发。

**如果是**: 需要评估维护成本是否值得。

---

### Q5: 是否属于MVP？

**定义**: 这个功能是否是MVP必须的？

**判断标准**:
- 是否在MVP范围内（Legacy系统、转生系统、世界模拟）？
- 是否影响核心循环的验证？
- 是否可以推迟到下一版本？

**如果否**: 推迟到下一版本。

## 审批流程

```
提出功能需求
    ↓
填写功能审批表
    ↓
回答5个审批问题
    ↓
┌─────────────────────────────────────┐
│ Q1 = 否 → 直接拒绝                   │
│ Q1 = 是 → 继续                       │
│ Q2 = 否 → 重新设计                    │
│ Q2 = 是 → 继续                       │
│ Q3 = 否 → 不影响开发，降低优先级       │
│ Q3 = 是 → 继续                       │
│ Q4 = 高 → 评估是否值得                │
│ Q4 = 中/低 → 继续                    │
│ Q5 = 否 → 推迟到下一版本              │
│ Q5 = 是 → 批准开发                    │
└─────────────────────────────────────┘
    ↓
批准/拒绝/推迟
```

## 功能审批表模板

```markdown
## 功能名称: [名称]

### Q1: 是否强化 Legacy Loop？
- 回答: [是/否]
- 说明: [如何强化]

### Q2: 玩家是否能感知？
- 回答: [是/否]
- 说明: [如何感知]

### Q3: 是否能在Steam截图中体现？
- 回答: [是/否]
- 说明: [如何体现]

### Q4: 是否增加维护成本？
- 回答: [高/中/低]
- 说明: [增加什么维护]

### Q5: 是否属于MVP？
- 回答: [是/否]
- 说明: [为什么是/不是]

### 审批结果: [批准/拒绝/推迟]
### 备注: [其他说明]
```

## 示例

### 示例1: Legacy可视化系统

```markdown
## 功能名称: Legacy可视化系统

### Q1: 是否强化 Legacy Loop？
- 回答: 是
- 说明: 玩家可以看到自己的行为如何改变世界

### Q2: 玩家是否能感知？
- 回答: 是
- 说明: 地图变化、NPC对话变化、建筑变化

### Q3: 是否能在Steam截图中体现？
- 回答: 是
- 说明: 可以展示"之前/之后"的对比截图

### Q4: 是否增加维护成本？
- 回答: 中
- 说明: 需要为每个Legacy类型设计可视化

### Q5: 是否属于MVP？
- 回答: 是
- 说明: 核心卖点，必须有

### 审批结果: 批准
```

### 示例2: 复杂战斗系统

```markdown
## 功能名称: 复杂战斗系统

### Q1: 是否强化 Legacy Loop？
- 回答: 否
- 说明: 战斗系统不直接影响Legacy

### Q2: 玩家是否能感知？
- 回答: 是
- 说明: 战斗是游戏的一部分

### Q3: 是否能在Steam截图中体现？
- 回答: 是
- 说明: 可以展示战斗画面

### Q4: 是否增加维护成本？
- 回答: 高
- 说明: 需要大量平衡性和测试

### Q5: 是否属于MVP？
- 回答: 否
- 说明: 可以用简单战斗代替

### 审批结果: 推迟到下一版本
```
