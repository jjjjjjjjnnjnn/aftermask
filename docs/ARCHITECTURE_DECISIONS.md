## ADR-001: 选择 Godot 4 引擎

**日期**: 2026-06-18
**状态**: 已批准

**背景**: 需要选择游戏引擎

**决策**: 使用 Godot 4.x + GDScript

**理由**:
- 单人开发效率最高
- 开源免费，无授权费
- GDScript上手快
- 本地AI集成方便（llama.cpp FFI）
- Steam发布成熟

**后果**:
- 未来性能关键路径可能需要Rust模块（通过FFI）
- GDScript性能约为C#的1/10
- 需要学习Godot特定的节点系统

**替代方案**:
- Unity: 生态成熟但收费模式争议，对独立开发者较重
- Unreal: 3A级画质但过重
- Bevy (Rust): 性能好但生态不成熟

## ADR-002: Legacy 优先于 Memory

**日期**: 2026-06-18
**状态**: 已批准

**背景**: 需要确定核心系统优先级

**决策**: Legacy系统优先于Memory系统

**理由**:
- 玩家感知更强（改变世界 > 被记住）
- 核心卖点更清晰（Every life leaves a mark）
- 全球玩家更容易理解
- 更强的轮回动力

**后果**:
- Memory系统降级为辅助系统
- 需要先实现Legacy系统
- 架构需要重新调整优先级

**替代方案**:
- Memory优先: 更容易实现，但玩家感知弱
- 平行开发: 资源分散，风险高

## ADR-003: 买断制（B2P）

**日期**: 2026-06-18
**状态**: 已批准

**背景**: 需要确定商业模式

**决策**: Steam买断制，不定价

**理由**:
- 国际核心玩家排斥F2P
- 买断制更符合模拟经营类游戏
- 避免P2W争议
- 定价在产品完成度确定后再决定

**后果**:
- 需要一次性提供完整体验
- 不能通过内购获取额外收入
- 需要更好的Steam商店页面

**替代方案**:
- F2P: 国内市场更大，但国际核心玩家排斥
- 订阅制: 不适合独立游戏
- 早期定价: 过早，风险高

## ADR-004: 纯文本原型先行

**日期**: 2026-06-18
**状态**: 已批准

**背景**: 需要验证核心乐趣

**决策**: 先做纯文本原型，再写代码

**理由**:
- 核心循环未验证
- 避免过早工程化
- 快速迭代成本低
- 可以直接验证玩家反馈

**后果**:
- 延迟了代码开发
- 可能发现核心循环不成立
- 需要找到真实玩家试玩

**替代方案**:
- 直接写代码: 风险高，可能浪费大量时间
- 做GDD文档: 已经做了很多，但文档不等于验证

## ADR-005: 机制国际化，文化中国化

**日期**: 2026-06-18
**状态**: 已批准

**背景**: 需要确定全球化策略

**决策**: 核心机制全球化，文化包装中国化

**理由**:
- 全球概念（轮回、遗产、后果）更容易理解
- 中国美学提供差异化
- 避免堆砌武侠术语
- 参考黑神话：悟空的成功路径

**后果**:
- 需要英文本地化P0
- 需要解释中国元素
- 可能失去部分纯中国玩家

**替代方案**:
- 纯中国风: 限制国际受众
- 纯西方风: 失去差异化
- 无文化特色: 失去独特性

## ADR-006: ECS 思想但不过早引入

**日期**: 2026-06-18
**状态**: 已批准

**背景**: 需要确定架构模式

**决策**: 采用ECS思想，但不过早引入完整ECS框架

**理由**:
- ECS适合大量实体同质行为
- 但当前阶段实体数量有限
- 过早引入增加复杂度
- 可以用Godot节点系统模拟

**后果**:
- 未来可能需要重构
- 但当前开发效率更高
- 预留了ECS迁移接口

**替代方案**:
- 完整ECS框架: 过早，复杂度高
- 纯OOP: 不适合大量实体
- Godot节点系统: 当前够用

## ADR-007: JSON 存档格式

**日期**: 2026-06-18
**状态**: 已批准

**背景**: 需要确定存档格式

**决策**: 使用JSON作为存档格式

**理由**:
- 跨平台兼容
- 人类可读，便于调试
- Godot内置JSON支持
- 未来可迁移到二进制格式

**后果**:
- 100世后存档可能较大
- 需要压缩和归档机制
- 解析速度比二进制慢

**替代方案**:
- 二进制格式: 更快，但不可读
- Godot Resource: 引擎特定，跨平台问题
- SQLite: 过重

## ADR-008: 纸面原型 → Godot原型 → 正式开发

**日期**: 2026-06-18
**状态**: 已批准

**背景**: 需要确定开发流程

**决策**: 三阶段开发流程

**理由**:
- 纸面原型验证核心乐趣（低成本）
- Godot原型验证技术可行性（中成本）
- 正式开发（高成本）
- 每阶段有明确的通过/失败标准

**后果**:
- 延迟了正式开发
- 但降低了整体风险
- 每阶段可以决定是否继续

**替代方案**:
- 直接开发: 风险高
- 无限原型: 范围失控
- 无验证直接发布: 失败率高
